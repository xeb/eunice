# eunice.xeb.ai — gemmad-backed webapp, gated to xebxeb@gmail.com

**Date:** 2026-07-22
**Status:** Design approved, pending spec review
**Goal:** Stand up `https://eunice.xeb.ai` as an always-on install of `eunice --webapp`
using the local `gemmad` daemon as its model, reachable only by `xebxeb@gmail.com`, with a
scheduled agent that emails a daily system-health report to `xebxeb@gmail.com`.

## Summary

`eunice.xeb.ai` fronts a local `eunice --webapp` process (systemd user service) that talks
to the already-running `gemmad` Gemma-4 daemon. Public reachability is delivered by the
existing `cloudflared` tunnel; access is restricted to `xebxeb@gmail.com` by a Cloudflare
Access application. This mirrors the existing `axxess.xeb.ai` deployment (also a eunice
webapp behind Access) and the `auth.xeb.ai` / `world.xeb.ai` tunnel-hosted, Access-gated
pattern documented in `~/p/master/docs/cloudflare.md`.

## Architecture

```
browser
  → eunice.xeb.ai              Cloudflare edge (Universal SSL covers first-level *.xeb.ai)
  → Cloudflare Access gate     allow: xebxeb@gmail.com only  (self-hosted app, 24h session)
  → cloudflared tunnel         f9a2e2e8-5e9e-4fa3-a99f-4a201a419e26 (already running)
  → eunice --webapp --gemmad   bound to 127.0.0.1:8812  (systemd user service: eunice.service)
  → gemmad                     127.0.0.1:18082  (already running: gemmad.service)
```

## Security model (load-bearing)

Eunice's webapp exposes a **full Bash shell and file read/write with no authentication of
its own**. `cloudflare.md` explicitly warns about this for the sibling `axxess.xeb.ai`
deployment: *"the origin (eunice webapp) has no auth of its own, so the Access app is the
only thing standing between the public and shell-as-…-user."* Therefore **two** controls are
mandatory and non-negotiable:

1. **Cloudflare Access** — a self-hosted Access app on `eunice.xeb.ai` with an allow policy
   for `xebxeb@gmail.com` only. This is the sole authentication layer.
2. **Loopback origin bind** — the webapp binds `127.0.0.1:8812`, **not** eunice's default
   `0.0.0.0`. This guarantees no host on the LAN can reach the shell un-gated; only the
   on-box `cloudflared` process can connect.

**Ordering rule (prevents a brief public shell):** create the Access application **before**
the `eunice` DNS record is published. Never publish the hostname while the origin is
reachable but ungated.

`gemmad`'s own `0.0.0.0:18082` LAN binding is out of scope — it is Bearer-key protected and
serves only model inference, not a shell.

## Components

### 1. systemd user service (`eunice.service`)

Generated by `eunice --install`, then edited for gemmad routing and start-ordering.

- **Install command** (run in a shell, so use an **absolute** path — `%h` only expands
  inside unit directives, not on the command line):
  `eunice --webapp --host 127.0.0.1 --port 8812 --agents /home/xeb/.eunice/webapp/agents.toml --install`
  (`--install` writes `~/.config/systemd/user/eunice.service`, **validates the agents file**,
  snapshots env into `~/.eunice/eunice.env`, runs `daemon-reload`, `enable`, `restart`, and
  `enable-linger`. The agents.toml must exist and be valid before install. The absolute path
  is baked verbatim into `ExecStart`.)
- **Post-install unit edits** (`eunice --install` does not emit these):
  - Add `--gemmad` to `ExecStart` so the webapp resolves gemmad as its model at startup.
    (`--install` only bakes `--port/--host/--model/--agents/--prompt/--no-persist`; the
    webapp only probes gemmad when `--gemmad` is present, and resolves its model once at
    process start.)
  - Add `After=gemmad.service` and `Wants=gemmad.service` under `[Unit]`. gemmad can take up
    to 180 s to load the model into VRAM; combined with the unit's existing
    `Restart=on-failure` / `RestartSec=5`, eunice retries until gemmad is reachable. (With
    `--gemmad`, if gemmad is down at startup eunice exits non-zero rather than falling back
    to a cloud model — the retry loop is intentional and correct.)
- **Persistence:** default (persist). Sessions stored in `sessions.db` under the service
  `WorkingDirectory`. Set `WorkingDirectory=%h/.eunice/webapp` (created beforehand) so the
  DB — and the agent's shell cwd — live in a stable, dedicated location.
- **gemmad connection env** (the interactive shell running `--install` does not have these
  set, so add them to `~/.eunice/eunice.env`): `GEMMAD_HOST=127.0.0.1`, `GEMMAD_PORT=18082`,
  and `GEMMAD_KEYS_FILE=/home/xeb/.config/gemmad/keys.toml` (eunice reads the Bearer token
  from that file, or from `GEMMAD_API_KEY`). **Absolute paths** — `EnvironmentFile` values
  are literal and not subject to `%h` specifier expansion. Verify the token resolves. (If a
  path in the unit's own directives is used instead, `%h` is valid there — e.g.
  `WorkingDirectory=%h/.eunice/webapp`.)
- **Verification before exposure:** `curl -s http://127.0.0.1:8812/` returns the app, and a
  test chat reaches gemmad — confirmed **before** any tunnel/DNS/Access change.

### 2. cloudflared tunnel ingress

- Back up `~/.cloudflared/config.yml` (timestamped: `config.yml.bak.YYYYMMDD-HHMMSS`, per
  existing convention).
- Add, above the `- service: http_status:404` catch-all:
  ```yaml
  - hostname: eunice.xeb.ai
    service: http://localhost:8812
  ```
- `systemctl --user restart cloudflared.service`.

### 3. DNS

- Create a **proxied** CNAME `eunice` → `f9a2e2e8-5e9e-4fa3-a99f-4a201a419e26.cfargotunnel.com`
  in zone `xeb.ai` (`91b65806dd3c138bf2f417778bc01431`), via a `Zone:DNS:Edit` token.
- Recreates the record deleted 2026-07-06 (was an A record to a since-retired VPS
  `31.220.49.75`; the tunnel CNAME replaces it).
- No ACME `/.well-known` bypass app is needed — that gotcha is specific to Cloudflare Pages
  custom-domain HTTP validation. Universal SSL already covers first-level `*.xeb.ai`.

### 4. Cloudflare Access application

- Token: the verified-working v3 Apps&Policies token from `cloudflare.md`
  (`Account:Access: Apps and Policies`).
- Create self-hosted app: `{"name":"Eunice","domain":"eunice.xeb.ai","type":"self_hosted","session_duration":"24h"}`.
- Attach allow policy (Mark-only):
  `{"name":"Mark only","decision":"allow","include":[{"email":{"email":"xebxeb@gmail.com"}}],"session_duration":"24h","precedence":1}`.
- Team domain: `kockerbeck.cloudflareaccess.com` (existing).
- **Created before the DNS record is published** (see ordering rule above).

### 5. Scheduled agent: daily system-health email

A single agent in `~/.eunice/webapp/agents.toml`, run by the webapp's in-process scheduler
(`--agents`), emailing a health report to `xebxeb@gmail.com` once a day.

- **Schedule:** `0 7 * * *` (07:00 daily, server local time). Standard 5-field Unix cron, as
  `agents.toml` requires. Single day-pattern, so the day-field-intersection gotcha does not
  apply. Editable live in the web UI (v1.0.8 agent editor).
- **Send mechanism:** `gmail-cli` (present at `~/.local/bin/gmail-cli` and `~/.cargo/bin`,
  authenticated as `xebxeb@gmail.com`, token kept alive by `authd`). Send with
  `gmail-cli compose --to xebxeb@gmail.com --subject "…" --body-file <tmpfile>`.
- **PATH requirement:** the agent runs tools under the systemd user service, so
  `~/.eunice/eunice.env`'s snapshotted `PATH` must include `~/.local/bin` and `~/.cargo/bin`
  or the Bash tool won't find `gmail-cli`. Verify after install (`journalctl` + a manual
  "run now" from the web UI); add the paths to `eunice.env` if missing.
- **Prompt (deterministic, sized for gemma-4-12b):** instruct the agent to collect a fixed
  set of health facts via Bash and email a concise summary. Facts:
  - `hostname`, `uptime`, load average
  - disk: `df -h /` and the GreyArea mount
  - memory: `free -h`
  - key services: `systemctl --user is-active gemmad cloudflared eunice authd`
  - failed units: `systemctl --user --failed`
  - gemmad reachable: `curl -s -o /dev/null -w '%{http_code}' http://127.0.0.1:18082/v1/models`
  - GPU: `nvidia-smi --query-gpu=memory.used,memory.total,temperature.gpu --format=csv`
    (best-effort; note if unavailable)

  Then: write the summary to a temp file and send it via the exact `gmail-cli compose`
  command above, subject like `eunice health — <hostname> <date>`. Keep the body short and
  scannable; lead with an overall OK / ATTENTION verdict.
- **Idempotent/no side effects beyond the email.** The agent only reads system state and
  sends one message; it must not modify services or files outside its temp file.

## Data flow

1. Browser hits `eunice.xeb.ai`; Cloudflare Access intercepts, requires Google login as
   `xebxeb@gmail.com` (302 to the Access login on first visit).
2. On success, the edge forwards through the tunnel to `127.0.0.1:8812`.
3. eunice serves the webapp; each chat turn calls gemmad at `127.0.0.1:18082` with the
   Bearer token, running tools (Bash/Read/Write/Skill) locally under the service user.
4. Sessions persist to `~/.eunice/webapp/sessions.db`.
5. Independently, the in-process scheduler fires the `daily-health-email` agent at 07:00,
   running it as a tagged agent session that collects host facts and sends one email via
   `gmail-cli` to `xebxeb@gmail.com`.

## Error handling / edge cases

- **gemmad down at boot:** `After=/Wants=gemmad.service` + `Restart=on-failure` → eunice
  retries every 5 s until gemmad is ready.
- **gemmad restarts mid-session:** eunice reconnects on the next turn (existing client retry
  logic); no eunice restart needed since the model is resolved per process, not per call,
  and the model id is stable.
- **Access misconfig / gate open:** verified explicitly — `curl -I https://eunice.xeb.ai`
  must return `302` to `…cloudflareaccess.com` before the deployment is considered done.
- **LAN bypass:** prevented by the `127.0.0.1` bind; verified with
  `ss -tlnp | grep 8812` showing `127.0.0.1:8812`, not `0.0.0.0:8812`.

## Verification checklist (done = all pass)

1. `ss -tlnp | grep 8812` → bound to `127.0.0.1` only.
2. `curl -s http://127.0.0.1:8812/` → app loads on-box.
3. Local chat turn → response demonstrably from gemmad (Gemma), tools execute.
4. `curl -I https://eunice.xeb.ai` → `302` to `kockerbeck.cloudflareaccess.com` (gate live).
5. Browser, logged in as `xebxeb@gmail.com` → app loads, chat works end-to-end.
6. `systemctl --user is-enabled eunice.service` → `enabled`; survives a `systemctl --user
   restart eunice.service` and reconnects to gemmad.
7. Update `cloudflare.md` "Tunnel-hosted sites" + "Active Access Applications" tables with
   the new `eunice.xeb.ai` app id.
8. Trigger the health agent once (web UI "run now" or wait for the cron) → an email arrives
   at `xebxeb@gmail.com` with real host facts; the run shows success in the web UI.

## Out of scope (YAGNI)

- A custom webapp system prompt (`--prompt`) and additional agents beyond the daily
  health-email one.
- Changing gemmad's `0.0.0.0:18082` LAN binding.
- Any auth logic inside eunice itself (Access is the boundary, by design).
- HTML-formatted email — plain text is enough for a health report.
